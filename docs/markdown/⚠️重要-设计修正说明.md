## 💡 正确的业务逻辑

### 场景：修改深圳社保的上下限

```
前端操作：
社保管理 → 地区列表 → 选择"深圳" → 修改上下限 3000-30000

后端处理：
1. 更新 social_security_regions 表的上下限
2. 触发补差计算
3. 查找所有使用深圳社保的员工
4. 为每个员工生成补差记录
```

### 影响范围

```
修改深圳社保上下限 → 影响所有使用深圳社保的员工
- 员工A（养老保险）
- 员工B（养老+医疗）
- 员工C（养老+医疗+失业）
```

所有使用深圳社保的员工都会生成补差记录。
