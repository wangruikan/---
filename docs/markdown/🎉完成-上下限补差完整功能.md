## 📋 核心逻辑

### 1. 新旧上下限记录
```
第1次设置：2360-26421
├─ new_min_base_amount = 2360
├─ new_max_base_amount = 26421
└─ old字段为NULL（不补差）

第2次修改：3000-30000
├─ old_min_base_amount = 2360（复制自第1次的new）
├─ old_max_base_amount = 26421
├─ new_min_base_amount = 3000
├─ new_max_base_amount = 30000
└─ 生成补差记录

第3次修改：3500-35000（同月）
├─ old_min_base_amount = 3000（复制自第2次的new）
├─ old_max_base_amount = 30000
├─ new_min_base_amount = 3500
├─ new_max_base_amount = 35000
└─ 更新补差记录
```

### 2. 基数读取逻辑
```
旧基数：查找本月之前最近一次已生效的调基记录
新基数：查找本月是否有已生效的调基记录
       - 有：使用本月的基数
       - 没有：使用旧基数（但会用新上下限约束）
```

### 3. 补差计算
```
旧基数（用旧上下限约束）vs 新基数（用新上下限约束）

例如：
旧基数 2800（用2360-26421约束）→ 2800
新基数 2800（用3000-30000约束）→ 3000（低于下限）

补差 = 3000 - 2800 = 200
金额 = 200 × 比例 × 9个月
```

### 4. 补差期间
```
固定从今年1月到修改上下限的前一个月

例如：10月修改 → 补差 2025-01 至 2025-09（9个月）
```

### 5. 数据存储
```
补差记录表（insurance_compensation_records）：
- old_base：直接存储约束后的旧基数
- new_base：直接存储约束后的新基数  
- compensation_details：JSON数组，按保险类型分组
  社保：[{name: "养老保险", ...}, {name: "失业保险", ...}]
  公积金：[{name: "标准配置", ...}]
```
