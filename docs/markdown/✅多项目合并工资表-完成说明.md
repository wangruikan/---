# ✅ 多项目合并工资表功能 - 完成说明

## 📋 功能概述

实现了在生成工资表时，可以选择多个项目，系统会将所有选中项目的员工合并到**一条工资表记录**中。

## 🎯 核心功能

### 1. 多项目选择
- 前端支持多选项目（带标签折叠）
- 可以清晰看到已选择的项目数量
- 显示每个项目的考勤状态（无需考勤/考勤未审批）

### 2. 合并生成逻辑
- **一次提交，一条记录**：选择N个项目，生成1条工资表记录
- **主项目ID**：使用第一个选中的项目ID作为工资表的 `project_id`
- **员工去重**：自动去重，同一员工只计算一次
- **部门保留**：每个员工的部门名称保留为其实际所属项目名称

### 3. 考勤检查
- 检查所有选中项目的考勤状态
- 只要有一个项目需要考勤且未审批，就不能生成
- 如果项目设置为"无需考勤"，则跳过该项目的考勤检查

## 📝 修改的文件

### 后端
- **`app/Http/Controllers/SalaryController.php`**
  - `generate()` 方法接收 `project_ids` 数组
  - 获取所有选中项目的员工并去重
  - 检查所有项目的考勤状态
  - 所有员工使用主项目ID创建工资记录
  - 每个员工的部门保留为其实际所属项目名称

### 前端
- **`src/views/Salaries/index.vue`**
  - 项目选择器支持多选
  - 已选项目数量提示
  - 提交 `project_ids` 数组到后端

## 🔄 业务流程

```
1. 用户选择多个项目（如：公园项目、老年公寓）
   ↓
2. 前端提交：{ month: '2024-11', project_ids: [1, 2] }
   ↓
3. 后端检查：
   - 公园项目：需要考勤 → 检查是否已审批 ✅
   - 老年公寓：无需考勤 → 跳过检查 ⏭️
   ↓
4. 后端获取员工：
   - 公园项目：10人
   - 老年公寓：15人
   - 去重后：25人
   ↓
5. 后端生成工资记录：
   - project_id: 1 (主项目ID)
   - 25条员工工资记录
   - 每条记录的 department 保留原项目名称
   ↓
6. 前端显示：成功生成工资表（包含 2 个项目：公园项目、老年公寓）
```

## 💾 数据结构示例

### 请求
```json
{
  "month": "2024-11",
  "project_ids": [1, 2, 3]
}
```

### 响应
```json
{
  "success": true,
  "message": "成功生成 2024-11 的工资表，共 50 条记录（来自 3 个项目）",
  "data": {
    "count": 50,
    "project_id": 1,
    "project_ids": [1, 2, 3],
    "project_names": ["公园项目", "老年公寓", "物业管理"],
    "month": "2024-11"
  }
}
```

### 生成的工资记录
```
| id | project_id | employee_id | department       | employee_name | ... |
|----|------------|-------------|------------------|---------------|-----|
| 1  | 1          | 101         | 公园项目         | 张三          | ... |
| 2  | 1          | 102         | 公园项目         | 李四          | ... |
| 3  | 1          | 201         | 老年公寓         | 王五          | ... |
| 4  | 1          | 202         | 老年公寓、物业管理 | 赵六         | ... |
| 5  | 1          | 301         | 物业管理         | 孙七          | ... |
```

**注意事项**：
- 所有记录的 `project_id` 都是主项目ID（第一个选中的项目）
- `department` 字段保留了员工的实际所属项目名称
- 如果员工属于多个选中的项目（如赵六），部门名称会用顿号（、）连接
- 员工只会生成一条工资记录，即使属于多个选中的项目

## ✅ 测试场景

### 场景1：单项目生成（保持向后兼容）
- 选择1个项目
- 行为与之前完全一致

### 场景2：多项目生成（都无需考勤）
- 选择2个项目，都设置为"无需考勤"
- 直接合并生成

### 场景3：多项目生成（混合考勤要求）
- 项目A：需要考勤，已审批 ✅
- 项目B：无需考勤 ⏭️
- 成功合并生成

### 场景4：多项目生成（有未审批）
- 项目A：需要考勤，已审批 ✅
- 项目B：需要考勤，未审批 ❌
- 提示错误，不能生成

### 场景5：员工去重
- 项目A有员工：张三、李四、王五
- 项目B有员工：王五、赵六（王五在两个项目）
- 最终只为王五生成一条记录

## 🎉 优势

1. **简化操作**：多个项目一次搞定，不需要分别生成
2. **数据整合**：所有员工在一张工资表里，方便统一管理
3. **部门清晰**：虽然合并了，但每个员工的部门仍然能看出是哪个项目
4. **向后兼容**：单项目生成的功能完全不受影响
5. **灵活考勤**：支持部分项目需要考勤、部分不需要的混合场景

## 📌 注意事项

1. **主项目ID**：工资表的 `project_id` 使用第一个选中的项目ID
2. **员工数据结构**：员工表的 `project_ids` 是 JSON 数组，一个员工可以属于多个项目
3. **员工查询**：使用 `whereJsonContains` 查询所有属于选中项目的员工
4. **员工去重**：如果同一员工在多个选中项目中，只会生成一条工资记录
5. **部门名称**：如果员工属于多个选中项目，部门名称用顿号（、）连接
6. **考勤检查**：所有需要考勤的项目都必须审批通过
7. **保险设置**：使用主项目的保险导入设置
8. **重复检查**：检查主项目ID + 月份是否已存在工资表
