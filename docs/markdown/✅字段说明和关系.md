# ✅ 字段说明和关系

## 📊 social_security_regions 表的字段

### 1. 原始字段（必须保留）
```
min_base_amount  - 最低基数
max_base_amount  - 最高基数
```

**用途**：
- ✅ 参保人员基数约束（`InsurancePersonnel.php`）
- ✅ 参保明细生成基数约束（`InsuranceChangeController.php`）
- ✅ 前端显示当前上下限

**更新时机**：
- 每次修改上下限时，同步更新为最新值

### 2. 旧上下限字段（补差用）
```
old_min_base_amount     - 旧最低基数
old_max_base_amount     - 旧最高基数
old_limits_updated_at   - 旧上下限修改时间
```

**用途**：
- ✅ 补差计算时，约束旧基数

**更新时机**：
- 第1次设置：NULL
- 本月修改：不变
- 跨月修改：复制自new字段

### 3. 新上下限字段（补差用）
```
new_min_base_amount     - 新最低基数
new_max_base_amount     - 新最高基数
new_limits_updated_at   - 新上下限修改时间
```

**用途**：
- ✅ 补差计算时，约束新基数
- ✅ 判断是否本月修改

**更新时机**：
- 每次修改上下限时更新

## 🔄 字段更新流程

### 修改上下限时
```
用户输入：3000-30000
      ↓
BaseLimitUpdateService：
  1. 判断是否需要复制到old
  2. 更新new字段 = 3000-30000
  3. 同步更新原字段 = 3000-30000
      ↓
结果：
  min_base_amount = 3000      ← 同步更新
  max_base_amount = 30000     ← 同步更新
  new_min_base_amount = 3000  ← 直接更新
  new_max_base_amount = 30000 ← 直接更新
  old_min_base_amount = 2360  ← 条件更新
  old_max_base_amount = 26421 ← 条件更新
```

## 📝 各字段的职责

### `min_base_amount` / `max_base_amount`
- **职责**：日常业务使用（约束基数、前端显示）
- **特点**：始终是当前生效的上下限
- **能否删除**：❌ 不能删除，很多地方在用

### `new_min_base_amount` / `new_max_base_amount`
- **职责**：补差计算使用（新上下限）
- **特点**：记录最新修改的值
- **能否删除**：❌ 不能删除，补差必需

### `old_min_base_amount` / `old_max_base_amount`
- **职责**：补差计算使用（旧上下限）
- **特点**：记录上一次的值
- **能否删除**：❌ 不能删除，补差必需

## 🎯 结论

**所有字段都必须保留！**

- `min/max_base_amount` - 日常业务使用
- `new_min/max_base_amount` - 补差使用（新值）
- `old_min/max_base_amount` - 补差使用（旧值）

它们的关系是：
```
min_base_amount = new_min_base_amount（同步）
max_base_amount = new_max_base_amount（同步）
```

## ✅ 当前实现

已在 `BaseLimitUpdateService.php` 中实现了同步更新：
- ✅ 每次更新 `new_*` 字段时，同步更新 `min/max_base_amount`
- ✅ 保证两组字段始终一致
- ✅ 满足日常业务和补差计算的双重需求

这样设计既保证了兼容性，又满足了补差需求！🎉
