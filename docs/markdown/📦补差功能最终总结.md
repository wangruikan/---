### 2. ✅ 统一的筛选逻辑（按创建日期）

**参保增减**和**工资管理**现在使用相同的筛选逻辑。

#### 筛选规则
```php
// 按创建日期筛选
WHERE YEAR(created_at) = 2025
  AND MONTH(created_at) = 10
```

#### 业务含义
- **选择/生成 2025-10 月** → 显示在 2025-10 **创建**的补差记录
- **选择/生成 2025-09 月** → 显示在 2025-09 **创建**的补差记录

#### 为什么这样设计？
1. **会计核算**：补差在创建月份计入当月工资
2. **逻辑一致**：参保增减和工资管理使用相同规则
3. **便于追溯**：按创建时间查看补差历史

### 3. ✅ 月份自动触发查询

在参保增减页面，选择月份时自动查询，无需点击"查询"按钮。

#### 实现方式
```javascript
watch(() => detailFilterForm.value.month, (newMonth, oldMonth) => {
  if (newMonth !== oldMonth && oldMonth !== undefined) {
    // 自动触发相应的加载函数
    loadCompensationDetails()  // 或其他函数
  }
})
```

#### 用户体验
- **修改前**：选择月份 → 点击查询 → 显示结果
- **修改后**：选择月份 → ✅ 自动显示结果

## 🎯 核心设计理念

### 统一的筛选逻辑

| 场景 | 筛选字段 | SQL条件 | 业务含义 |
|------|---------|---------|---------|
| **参保增减** | `created_at` | `YEAR(created_at) = 2025 AND MONTH(created_at) = 10` | 查看10月创建的补差记录 |
| **工资管理** | `created_at` | `YEAR(created_at) = 2025 AND MONTH(created_at) = 10` | 10月工资包含10月新增的补差 |

### 为什么按创建日期？

1. **会计逻辑**：
   - 补差在创建月份计入当月工资
   - 符合"权责发生制"原则

2. **实际场景**：
   ```
   2025-10-25：发现之前基数错误，创建补差记录
   → 补差应该体现在10月工资中
   → 10月工资发放时包含补差金额
   ```

3. **一致性**：
   - 参保增减和工资管理使用相同逻辑
   - 便于对账和核查
