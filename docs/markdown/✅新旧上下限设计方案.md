# ✅ 新旧上下限设计方案

## 🎯 设计思路

通过在保险类型表中添加 `old_*` 和 `new_*` 字段来记录上下限的新旧值，使用时间戳判断是否跨月修改。

## 📊 数据库字段设计

### 添加的字段

```sql
-- 旧上下限（上一次的值）
old_min_base_amount DECIMAL(10,2)
old_max_base_amount DECIMAL(10,2)
old_limits_updated_at TIMESTAMP

-- 新上下限（当前的值）
new_min_base_amount DECIMAL(10,2)
new_max_base_amount DECIMAL(10,2)
new_limits_updated_at TIMESTAMP
```

### 应用到的表

- `social_security_types` - 社保类型表
- `medical_insurance_types` - 医保类型表
- `housing_fund_configs` - 公积金配置表

## 🔄 逻辑流程

### 场景1：第一次设置上下限

```
当前状态：
├─ old_min_base_amount: NULL
├─ old_max_base_amount: NULL
├─ new_min_base_amount: NULL
├─ new_max_base_amount: NULL
└─ new_limits_updated_at: NULL

用户设置：2360-26421

操作：
├─ new_min_base_amount = 2360
├─ new_max_base_amount = 26421
├─ new_limits_updated_at = 2025-10-15
└─ ❌ 不生成补差（第一次设置，没有旧值）
```

### 场景2：第2次修改（生成补差）

```
当前状态：
├─ old_min_base_amount: NULL
├─ old_max_base_amount: NULL
├─ new_min_base_amount: 2360
├─ new_max_base_amount: 26421
└─ new_limits_updated_at: 2025-10-15

用户修改：3000-30000（当前时间：2025-10-20）

操作：
1. 将new复制到old：
   ├─ old_min_base_amount = 2360
   ├─ old_max_base_amount = 26421
   └─ old_limits_updated_at = 2025-10-15

2. 设置新的new值：
   ├─ new_min_base_amount = 3000
   ├─ new_max_base_amount = 30000
   └─ new_limits_updated_at = 2025-10-20

3. ✅ 生成补差记录
   ├─ 旧基数（旧上下限2360-26421约束）
   ├─ 当前基数（新上下限3000-30000约束）
   └─ 补差期间：2025-01 至 2025-09
```

### 场景3：第3次修改（更新补差）

```
当前状态：
├─ old_min_base_amount: 2360
├─ old_max_base_amount: 26421
├─ old_limits_updated_at: 2025-10-15
├─ new_min_base_amount: 3000
├─ new_max_base_amount: 30000
└─ new_limits_updated_at: 2025-10-20

用户修改：3500-35000（当前时间：2025-10-25，同月）

操作：
1. 将new复制到old（覆盖）：
   ├─ old_min_base_amount = 3000（覆盖2360）
   ├─ old_max_base_amount = 30000（覆盖26421）
   └─ old_limits_updated_at = 2025-10-20

2. 设置新的new值：
   ├─ new_min_base_amount = 3500
   ├─ new_max_base_amount = 35000
   └─ new_limits_updated_at = 2025-10-25

3. ✅ 更新补差记录（本月已有，更新它）
   ├─ 旧基数（旧上下限3000-30000约束）
   ├─ 当前基数（新上下限3500-35000约束）
   └─ 补差期间：2025-01 至 2025-09（不变）
```

### 场景4：跨月修改（生成新补差）

```
当前状态：
├─ old_min_base_amount: 3000
├─ old_max_base_amount: 30000
├─ old_limits_updated_at: 2025-10-20
├─ new_min_base_amount: 3500
├─ new_max_base_amount: 35000
└─ new_limits_updated_at: 2025-10-25

用户修改：4000-40000（当前时间：2025-11-15，跨月）

操作：
1. 将new复制到old（覆盖）：
   ├─ old_min_base_amount = 3500（覆盖3000）
   ├─ old_max_base_amount = 35000（覆盖30000）
   └─ old_limits_updated_at = 2025-10-25

2. 设置新的new值：
   ├─ new_min_base_amount = 4000
   ├─ new_max_base_amount = 40000
   └─ new_limits_updated_at = 2025-11-15

3. ✅ 生成新的补差记录（新月份）
   ├─ 旧基数（旧上下限3500-35000约束）
   ├─ 当前基数（新上下限4000-40000约束）
   └─ 补差期间：2025-01 至 2025-10（新期间）
```

## 🎯 核心优势

### ✅ 1. 不需要历史记录表
- 只保留最近两次的上下限
- 减少数据库表数量
- 查询效率高

### ✅ 2. 每次修改都生成/更新补差
- 第2次及以后的修改都会生成补差
- 本月已有补差记录时自动更新
- 跨月时生成新的补差记录

### ✅ 3. 自动判断创建或更新
- 按补差期间（起止月份）判断是否已存在
- 已存在则更新，不存在则创建
- 无需手动触发

### ✅ 4. 数据完整性
- 始终保留最近两次的完整数据
- 可追溯最近一次的变更
- 满足补差计算需求

## 🚀 使用示例

```php
use App\Services\BaseLimitUpdateService;
use App\Services\BaseLimitCompensationService;

// 1. 更新上下限
$updateService = app(BaseLimitUpdateService::class);
$result = $updateService->updateLimits($socialSecurityType, 3500, 35000);

// 2. 判断是否需要补差
if ($result['should_compensate']) {
    // 3. 计算补差
    $compensationService = app(BaseLimitCompensationService::class);
    $compensationService->calculateSocialSecurityCompensation(
        $socialSecurityType->region_id,
        $result['old_min'],  // 旧下限
        $result['old_max'],  // 旧上限
        $result['new_min'],  // 新下限
        $result['new_max']   // 新上限
    );
}
```

## 📊 数据示例

### 表结构示例（social_security_types）

| id | name | min_base_amount | max_base_amount | old_min | old_max | old_updated_at | new_min | new_max | new_updated_at |
|----|------|-----------------|-----------------|---------|---------|----------------|---------|---------|----------------|
| 1 | 养老保险 | 2360 | 26421 | 3000 | 30000 | 2025-10-20 | 3500 | 35000 | 2025-11-15 |

**说明**：
- `min_base_amount` / `max_base_amount` - 保留原字段（兼容性）
- `old_*` - 上一次的上下限（2025-10-20设置的）
- `new_*` - 当前的上下限（2025-11-15设置的）

## ⚠️ 注意事项

1. **迁移时的数据处理**
   - 现有的 `min_base_amount` 和 `max_base_amount` 会复制到 `new_*` 字段
   - `old_*` 字段初始为 NULL

2. **兼容性**
   - 保留原有的 `min_base_amount` 和 `max_base_amount` 字段
   - 前端可以继续使用这些字段显示当前值
   - 后端逻辑使用 `new_*` 字段

3. **时间判断**
   - 使用 `startOfMonth()` 比较月份
   - 确保同月判断的准确性
