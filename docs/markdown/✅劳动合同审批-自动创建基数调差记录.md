# ✅ 劳动合同审批 - 自动创建基数调差记录

## 功能说明

当新员工的劳动合同审批通过后，系统会自动执行以下操作：

1. **创建参保人员信息**（原有功能）
2. **自动创建基数调差记录**（新增功能）✨

## 业务流程

### 1️⃣ 新增员工
在人员档案管理中新增员工时，填写以下基数信息：
- 社保基数（`social_security_base`）
- 医保基数（`medical_insurance_base`）
- 公积金基数（`housing_fund_base`）
- 大额医疗基数（`large_medical_base`）

### 2️⃣ 添加劳动合同
在合同管理中为该员工添加劳动合同并提交审批

### 3️⃣ 审批通过后自动执行
系统会自动：

#### ✅ 创建参保人员信息
- 在保险增减管理中自动新增该员工
- 状态为"待处理"
- 包含员工的基本信息和基数

#### ✅ 创建基数调差记录（新功能）
- 在基数调差管理中自动新增该员工
- **状态为"已生效"**（直接生效）
- **旧基数为 0**（因为是新员工）
- **新基数为员工创建时填写的基数**
- **生效日期为当前日期**
- 调整原因：`劳动合同审批通过，自动导入员工基数`

## 技术实现

### 修改的文件
- `app/Services/ApprovalService.php`

### 核心方法

#### 1. `autoImportInsuranceInfo()` - 修改
在原有的自动导入保险信息方法中，新增调用基数调差创建方法：

```php
// 创建新的保险变更记录
$this->createNewInsuranceChange($employee, $project, $contract->account_set_id, $contract->created_by);

// 自动创建基数调差记录（将员工创建时填写的基数作为新基数，直接生效）
$this->autoCreateBaseAdjustment($employee, $contract->account_set_id, $contract->created_by);
```

#### 2. `autoCreateBaseAdjustment()` - 新增
自动创建基数调差记录的核心方法：

**功能特点**：
- ✅ 检查员工是否填写了基数（至少一项）
- ✅ 检查是否已存在待生效的调整记录（避免重复）
- ✅ 旧基数设为 0（新员工）
- ✅ 新基数为员工创建时填写的值
- ✅ 状态直接设为 `effective`（已生效）
- ✅ 生效日期为当前日期
- ✅ 记录生效时间 `applied_at`
- ✅ 详细的日志记录

**创建的数据结构**：

## 业务逻辑说明

### 为什么旧基数是 0？
因为是新员工，之前没有参保记录，所以旧基数为 0。

### 为什么直接生效？
新员工入职后，基数应该立即生效，不需要等待审批或手动确认。这样可以：
- ✅ 简化操作流程
- ✅ 避免遗漏
- ✅ 确保基数及时生效

### 如果员工没有填写基数怎么办？
- 系统会检查员工是否至少填写了一项基数
- 如果所有基数都为空或为 0，则不创建基数调差记录
- 日志会记录跳过原因

### 如果已存在待生效的调整记录怎么办？
- 系统会检查是否已存在 `status = 'pending'` 的记录
- 如果存在，则跳过创建，避免重复
- 日志会记录跳过原因和现有记录ID

## 使用场景

### 场景 1：标准新员工入职
1. HR 在系统中新增员工张三
2. 填写社保基数 5000、医保基数 5000、公积金基数 5000
3. 为张三添加劳动合同并提交审批
4. 审批通过后：
   - ✅ 自动在保险增减管理中创建张三的参保记录
   - ✅ 自动在基数调差管理中创建张三的基数记录（已生效）

### 场景 2：员工未填写基数
1. HR 在系统中新增员工李四
2. 没有填写任何基数信息
3. 为李四添加劳动合同并提交审批
4. 审批通过后：
   - ✅ 自动在保险增减管理中创建李四的参保记录
   - ⏭️ 不创建基数调差记录（因为没有基数）

### 场景 3：部分基数填写
1. HR 在系统中新增员工王五
2. 只填写了社保基数 5000 和医保基数 5000
3. 为王五添加劳动合同并提交审批
4. 审批通过后：
   - ✅ 自动创建基数调差记录
   - ✅ 只包含社保和医保基数
   - ⏭️ 公积金和大额医疗基数为空

## 注意事项

1. **基数来源**：基数来自员工档案中填写的值，不是从项目配置中读取
2. **状态直接生效**：创建的记录状态为 `effective`，不需要再次确认
3. **避免重复**：如果已存在待生效的记录，不会重复创建
4. **只针对劳动合同**：只有劳动合同审批通过才会触发，解除协议和退休协议不会触发
5. **日志追踪**：所有操作都有详细的日志记录，便于排查问题
