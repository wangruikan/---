执行以下SQL验证现有数据是否正确迁移：

```sql

SELECT 
    id,
    name,
    min_base_amount AS '原下限',
    max_base_amount AS '原上限',
    new_min_base_amount AS '新下限',
    new_max_base_amount AS '新上限'
FROM social_security_types
WHERE min_base_amount IS NOT NULL
LIMIT 5;
```

**预期结果**：
- `new_min_base_amount` 应该等于 `min_base_amount`
- `new_max_base_amount` 应该等于 `max_base_amount`
- `old_min_base_amount` 和 `old_max_base_amount` 应该是 NULL

## ⚠️ 注意事项

### 1. 备份数据库
在执行迁移前，请务必备份数据库：

```bash
mysqldump -u用户名 -p 数据库名 > backup_$(date +%Y%m%d_%H%M%S).sql
```

### 2. 测试环境先测试
建议先在测试环境执行，确认无误后再在生产环境执行。

### 3. 现有数据处理
- 现有的 `min_base_amount` 和 `max_base_amount` 不会被删除
- 它们会被复制到 `new_*` 字段
- 前端可以继续使用这些字段显示当前值

### 4. 第一次修改上下限
- 第一次修改上下限时不会生成补差（因为没有旧值）
- 从第二次修改开始才会生成补差

## 🧪 测试步骤

### 测试1：第一次设置上下限

1. 在前端修改某个社保类型的上下限
2. 保存后查看数据库：

```sql
SELECT 
    id,
    name,
    old_min_base_amount,
    old_max_base_amount,
    new_min_base_amount,
    new_max_base_amount,
    new_limits_updated_at
FROM social_security_types
WHERE id = 你修改的ID;
```

**预期结果**：
- `old_*` 字段为 NULL
- `new_*` 字段有值
- 不生成补差记录

### 测试2：第二次修改上下限

1. 再次修改同一个社保类型的上下限
2. 保存后查看数据库：

```sql
-- 查看上下限字段
SELECT 
    id,
    name,
    old_min_base_amount,
    old_max_base_amount,
    new_min_base_amount,
    new_max_base_amount
FROM social_security_types
WHERE id = 你修改的ID;

-- 查看补差记录
SELECT 
    employee_name,
    old_base,
    new_base,
    compensation_start_month,
    compensation_end_month,
    compensation_months,
    total_amount
FROM insurance_compensation_records
WHERE compensation_type = 'social_security'
ORDER BY created_at DESC
LIMIT 10;
```

**预期结果**：
- `old_*` 字段有值（第一次设置的值）
- `new_*` 字段有新值（第二次设置的值）
- 生成了补差记录

### 测试3：同月多次修改

1. 在同一个月内第三次修改上下限
2. 查看补差记录：

```sql
SELECT 
    employee_name,
    old_base,
    new_base,
    total_amount,
    updated_at
FROM insurance_compensation_records
WHERE compensation_type = 'social_security'
  AND DATE_FORMAT(updated_at, '%Y-%m') = DATE_FORMAT(NOW(), '%Y-%m')
ORDER BY updated_at DESC;
```

**预期结果**：
- 补差记录被更新（不是新增）
- `updated_at` 时间是最新的
