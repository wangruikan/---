# ✅ 修复确认参保时费率数据问题

## 🐛 问题描述

用户反馈：每次点击"确认处理"后，保险明细的金额又变成异常值（如 0.01, 0.11 等）。

## 🔍 问题根源

问题出现在**确认参保时的数据同步逻辑**中。费率数据在从数据库读取并转换为JSON格式时，没有显式转换为 `float` 类型，导致：

1. 费率被转换为字符串或其他格式
2. JSON 编码时保留了错误的数据类型
3. 计算金额时使用了错误的费率值

## 📋 技术说明

### 为什么需要显式转换？

在 Laravel 中，Model 的 `casts` 配置将费率定义为 `decimal:4`：

```php
protected $casts = [
    'employee_ratio' => 'decimal:4',  // 转换为字符串 "0.3003"
    'company_ratio' => 'decimal:4',
];
```

这会导致：
1. 数据库读取时，费率被转换为字符串格式（如 `"0.3003"`）
2. 在 PHP 中参与数学运算时，PHP 会自动类型转换，但...
3. 当编码为 JSON 时，可能保留字符串格式
4. 前端或后续处理可能无法正确识别

**解决方案**：在构建数组数据时，显式使用 `(float)` 强制转换，确保 JSON 编码时使用数值类型。

### JSON 编码的差异

```php
// 未转换
$data = ['ratio' => '0.3003'];
json_encode($data);  // {"ratio":"0.3003"}  ❌ 字符串

// 显式转换
$data = ['ratio' => (float)'0.3003'];
json_encode($data);  // {"ratio":0.3003}    ✅ 数值
```

## ⚠️ 预防措施

### 1. 代码规范

在任何构建费率数据的地方，都应该显式转换：

```php
✅ 正确：
'employee_ratio' => (float)$type->employee_ratio,

❌ 错误：
'employee_ratio' => $type->employee_ratio,
```

### 2. 单元测试

可以添加单元测试确保费率类型正确：

```php
public function test_insurance_ratio_types()
{
    $config = InsuranceChange::find(1)->getCurrentSocialSecurityConfig();
    
    foreach ($config as $type) {
        $this->assertIsFloat($type['employee_ratio']);
        $this->assertIsFloat($type['company_ratio']);
    }
}
```

### 3. 数据验证

在保存前验证费率值的合理性：

```php
if (!is_float($ratio) || $ratio < 0 || $ratio > 1) {
    throw new \Exception("费率值异常: " . var_export($ratio, true));
}
```
