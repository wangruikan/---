# ✅ 完成 - 上下限补差功能

## 🎉 功能概述

每次修改保险上下限时，自动生成/更新补差明细。

### 核心逻辑
```
修改上下限 
  ↓
查找旧基数和新基数
  ↓
应用各自的上下限约束
  ↓
计算补差并生成/更新明细
```

## 📝 功能说明

### 第1次设置上下限
```
操作：设置上下限 2360-26421
结果：
├─ new_min_base_amount = 2360
├─ new_max_base_amount = 26421
├─ new_limits_updated_at = 当前时间
└─ ❌ 不生成补差（没有旧值）
```

### 第2次修改上下限
```
操作：修改上下限 3000-30000
结果：
1. 复制new到old：
   ├─ old_min_base_amount = 2360
   ├─ old_max_base_amount = 26421
   └─ old_limits_updated_at = 第1次的时间

2. 设置新的new：
   ├─ new_min_base_amount = 3000
   ├─ new_max_base_amount = 30000
   └─ new_limits_updated_at = 当前时间

3. ✅ 生成补差明细：
   ├─ 查找旧基数：本月之前最近一次生效的基数
   ├─ 查找新基数：本月有调基就用本月的，没有就用旧基数
   ├─ 旧基数用旧上下限约束：2360-26421
   ├─ 新基数用新上下限约束：3000-30000
   ├─ 补差期间：今年1月 至 修改前一个月
   └─ 补差金额：(新约束后基数 - 旧约束后基数) × 比例 × 月数
```

### 第3次修改上下限（同月）
```
操作：同月再次修改 3500-35000
结果：
1. 复制new到old：
   ├─ old_min_base_amount = 3000（覆盖2360）
   ├─ old_max_base_amount = 30000（覆盖26421）
   └─ old_limits_updated_at = 第2次的时间

2. 设置新的new：
   ├─ new_min_base_amount = 3500
   ├─ new_max_base_amount = 35000
   └─ new_limits_updated_at = 当前时间

3. ✅ 更新补差明细（本月已有，更新它）：
   ├─ 更新旧基数用旧上下限约束：3000-30000
   ├─ 更新新基数用新上下限约束：3500-35000
   └─ 补差期间不变：今年1月 至 修改前一个月
```

## 🎯 基数读取逻辑

### 旧基数
```sql
-- 查找本月之前最近一次生效的基数
SELECT new_social_security_base 
FROM base_adjustments
WHERE employee_id = ?
  AND status = 'applied'
  AND social_security_effective_date IS NOT NULL
  AND social_security_effective_date < 当前月第一天
ORDER BY social_security_effective_date DESC
LIMIT 1
```

### 新基数
```sql
-- 查找本月是否有生效的基数
SELECT new_social_security_base 
FROM base_adjustments
WHERE employee_id = ?
  AND status = 'applied'
  AND social_security_effective_date IS NOT NULL
  AND YEAR(social_security_effective_date) = 当前年
  AND MONTH(social_security_effective_date) = 当前月
ORDER BY social_security_effective_date DESC
LIMIT 1

-- 如果没有，使用旧基数作为新基数
```

## 📊 补差计算示例

### 示例1：本月有调基
```
员工wrk：
├─ 2024-01-01：基数2800（已生效）
└─ 2025-10-01：基数3500（已生效）

当前：2025-10-15，修改上下限 2360-26421 → 3000-30000

计算：
├─ 旧基数：2800（本月之前最近一次）
│   └─ 用旧上下限约束：2360-26421 → 2800
├─ 新基数：3500（本月有调基）
│   └─ 用新上下限约束：3000-30000 → 3500
├─ 补差期间：2025-01 至 2025-09（9个月）
└─ 补差：(3500 - 2800) × 比例 × 9个月
```

### 示例2：本月没有调基
```
员工wrk：
└─ 2024-01-01：基数2800（已生效）

当前：2025-10-15，修改上下限 2360-26421 → 3000-30000

计算：
├─ 旧基数：2800（本月之前最近一次）
│   └─ 用旧上下限约束：2360-26421 → 2800
├─ 新基数：2800（本月没有调基，用旧基数）
│   └─ 用新上下限约束：3000-30000 → 3000（低于下限，用下限）
├─ 补差期间：2025-01 至 2025-09（9个月）
└─ 补差：(3000 - 2800) × 比例 × 9个月
```

## ⚠️ 注意事项

1. **医保不补差**
   - 医保类型虽然有新旧字段，但不会生成补差记录
   - `BaseLimitCompensationService::calculateMedicalInsuranceCompensation` 直接返回跳过

2. **补差期间**
   - 固定从今年1月到修改上下限的前一个月
   - 例如：10月修改，补差1-9月

3. **同月多次修改**
   - 补差记录会被更新，不会重复生成
   - 按补差期间（起止月份）判断是否已存在

4. **前端兼容性**
   - 保留了 `min_base_amount` 和 `max_base_amount` 字段
   - 前端可以继续使用这些字段显示当前值
