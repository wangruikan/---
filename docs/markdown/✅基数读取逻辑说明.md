# ✅ 基数读取逻辑说明

## 🎯 核心逻辑

修改上下限生成补差明细时，需要正确读取旧基数和新基数。

### 旧基数
```
查找：本月之前最近一次生效的基数
```

### 新基数
```
1. 先查找：本月是否有生效的基数
   - 如果有 → 使用本月的基数
   - 如果没有 → 使用旧基数（但会用新上下限约束）
```

## 📊 具体场景

### 场景1：本月有调基

```
员工wrk的调基记录：
├─ 2024-01-01：基数2800（已生效）
└─ 2025-10-01：基数3500（已生效）

当前时间：2025-10-15
修改上下限：2360-26421 → 3000-30000

查找旧基数：
├─ 查询条件：本月(2025-10)之前最近一次生效的基数
├─ 查询结果：2024-01-01，基数2800
└─ 旧基数 = 2800 ✅

查找新基数：
├─ 查询条件：本月(2025-10)是否有生效的基数
├─ 查询结果：有！2025-10-01，基数3500
└─ 新基数 = 3500 ✅

补差计算：
├─ 旧基数：2800
│   └─ 用旧上下限约束：2360-26421
│       └─ 2800在范围内 → 约束后：2800
├─ 新基数：3500
│   └─ 用新上下限约束：3000-30000
│       └─ 3500在范围内 → 约束后：3500
└─ 补差：3500 - 2800 = 700 ✅
```

## 💻 代码实现

### 查找旧基数

```php
// 1. 查找旧基数：本月之前最近一次生效的基数
$currentMonth = Carbon::now()->startOfMonth();

$lastAdjustmentBeforeThisMonth = BaseAdjustment::where('employee_id', $employee->id)
    ->where('status', 'applied')
    ->whereNotNull('social_security_effective_date')
    ->where('social_security_effective_date', '<', $currentMonth)  // 本月之前
    ->orderBy('social_security_effective_date', 'desc')
    ->first();

if ($lastAdjustmentBeforeThisMonth) {
    $oldBase = $lastAdjustmentBeforeThisMonth->new_social_security_base;
} else {
    // 如果没有历史调基记录，使用员工当前基数
    $oldBase = $employee->social_security_base;
}
```

### 查找新基数

```php
// 2. 查找新基数：本月是否有生效的基数
$thisMonthAdjustment = BaseAdjustment::where('employee_id', $employee->id)
    ->where('status', 'applied')
    ->whereNotNull('social_security_effective_date')
    ->whereYear('social_security_effective_date', Carbon::now()->year)
    ->whereMonth('social_security_effective_date', Carbon::now()->month)
    ->orderBy('social_security_effective_date', 'desc')
    ->first();

if ($thisMonthAdjustment) {
    // 本月有调基，使用本月的基数作为新基数
    $currentBase = $thisMonthAdjustment->new_social_security_base;
} else {
    // 本月没有调基，使用旧基数作为新基数（但会用新上下限约束）
    $currentBase = $oldBase;
}
```

### 应用约束并计算补差

```php
// 应用旧上下限约束到旧基数
$oldBaseConstrained = $this->applyConstraints($oldBase, $oldMinBase, $oldMaxBase);

// 应用新上下限约束到新基数
$newBaseConstrained = $this->applyConstraints($currentBase, $newMinBase, $newMaxBase);

// 如果约束后的基数不同，需要补差
if ($oldBaseConstrained != $newBaseConstrained) {
    // 生成补差记录
    $this->createCompensationRecord(...);
}
```

## 🎯 核心优势

### ✅ 1. 正确处理本月有调基的情况
- 旧基数：上月的基数
- 新基数：本月的基数
- 补差：反映了基数变化 + 上下限变化

### ✅ 2. 正确处理本月没有调基的情况
- 旧基数：上月的基数
- 新基数：上月的基数（但用新上下限约束）
- 补差：只反映上下限变化

### ✅ 3. 自动处理边界情况
- 没有历史调基记录：使用员工当前基数
- 本月多次调基：使用最新的调基记录
- 基数超出上下限：自动约束到范围内

### ✅ 4. 逻辑清晰简单
- 查询条件明确（本月之前 vs 本月）
- 代码易于理解和维护
- 符合实际业务需求
