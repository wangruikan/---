# ✅ 最终完整修复 - 基数约束功能

## 🎯 问题回顾

用户反馈：确认参保后，保险明细金额没有按照基数上下限约束计算，超过上限的基数仍按原值计算。

### 例子
- 员工社保基数：**5500**
- 地区上下限：**222 ~ 333**
- 期望：应按上限 **333** 计算
- 实际：仍按原值 **5500** 计算 ❌

---

## 🔍 问题根源分析

经过深度诊断，发现了**两个关键问题**：

### 问题1：约束逻辑从错误的地方读取上下限

**位置**：`app/Models/InsurancePersonnel::applyBaseConstraintsFromChange()`

**问题代码**（第265-279行，旧版本）：
```php
// 从类型的JSON快照中读取上下限
$firstType = reset($socialSecurityTypes);
$minBase = $firstType['min_base_amount'] ?? null;  // ❌ 类型中已无此字段
$maxBase = $firstType['max_base_amount'] ?? null;
```

**问题原因**：
- 基数上下限已从"类型级别"移到了"地区级别"
- 类型的JSON快照中不再包含 `min_base_amount` 和 `max_base_amount`
- 导致读取的上下限都是 `null`
- 约束逻辑无法工作

### 问题2：变更记录中地区ID为NULL

**现象**：
- 员工档案中有地区ID：`social_security_region_id = 1` ✅
- 变更记录中地区ID为：`social_security_region_id = NULL` ❌
- 参保记录中地区ID为：`social_security_region_id = NULL` ❌

**影响**：
- 即使修改了读取逻辑，如果变更记录中没有地区ID，也无法查询地区信息
- 导致无法获取上下限

## 🔧 完整修复方案

### 修复1：修改约束逻辑读取来源

**文件**：`app/Models/InsurancePersonnel.php`（第265-312行）

**修改内容**：从地区表读取上下限，并支持从员工档案回退

```php
// 修复后：从地区获取上下限，并支持回退到员工档案
$socialRegionId = $change->social_security_region_id;
if (!$socialRegionId && $change->employee) {
    $socialRegionId = $change->employee->social_security_region_id;  // 回退方案
}

if ($socialRegionId) {
    $socialRegion = \App\Models\SocialSecurityRegion::find($socialRegionId);
    if ($socialRegion) {
        $minBase = $socialRegion->min_base_amount;  // ✅ 从地区读取
        $maxBase = $socialRegion->max_base_amount;
        $socialSecurityBase = self::applyConstraint($socialSecurityBase, $minBase, $maxBase);
    }
}
```

### 修复2：添加基数约束到实时明细生成

**文件**：`app/Http/Controllers/InsuranceChangeController.php`

**方法1**：`generatePersonnelDetail()`（第389-433行）
**方法2**：`generateRealTimeDetail()`（第821-865行）

两个方法都添加了相同的约束逻辑：

```php
// 获取原始基数
$originalPensionBase = $personnel->employee_social_security_base ?? 0;

// 获取地区并应用约束
$socialRegion = \App\Models\SocialSecurityRegion::find($personnel->social_security_region_id);
$pensionBase = $this->applyBaseLimits(
    $originalPensionBase,
    $socialRegion ? $socialRegion->min_base_amount : null,
    $socialRegion ? $socialRegion->max_base_amount : null
);

// 使用约束后的基数计算金额
foreach ($socialSecurityTypes as $type) {
    $socialCompanyAmount += $pensionBase * $type['company_ratio'];
}
```

### 修复3：确保费率正确转换

**文件**：

## ✅ 修复效果

### 确认参保时的约束流程

```
1. 创建变更记录
   ↓
2. 确认参保（调用 getOrCreateFromInsuranceChange）
   ↓
3. applyBaseConstraintsFromChange 方法
   ├─ 从变更记录或员工档案获取地区ID
   ├─ 从地区表读取上下限
   └─ 应用约束：5500 > 333 → 使用 333 ✅
   ↓
4. 保存约束后的基数到 insurance_personnel 表
   ↓
5. 生成明细时使用约束后的基数
   ↓
6. 金额 = 333 × 费率 ✅
```

### 计算示例

**修复前**：
```
原始基数：5500
约束：无（上下限读取失败）
计算基数：5500
金额：5500 × 1.0 = 5500 ❌
```

**修复后**：
```
原始基数：5500
地区上下限：222 ~ 333
约束：5500 > 333 → 使用 333
计算基数：333
金额：333 × 1.0 = 333 ✅
```

## 📞 快速参考

### 约束规则
- 基数 < 下限 → 按下限
- 基数 > 上限 → 按上限
- 下限 ≤ 基数 ≤ 上限 → 按原值

### 上下限来源
- 社保：`social_security_regions` 表
- 医保：`medical_insurance_regions` 表
- 公积金：`housing_fund_params` JSON
- 大额医疗（按基数核算）：复用社保上下限
